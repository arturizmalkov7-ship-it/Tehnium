# CLAUDE.md — React best-practices + безопасность + UX

Используй этот документ как контракт качества для любых задач в этом репозитории.

## 1) Моя цель
- Дать простой, понятный и поддерживаемый код на React.
- Сфокусироваться на React best practices.
- Не допускать уязвимостей и небезопасных практик.
- Делать интерфейс удобным для пользователя (user-friendly).
- Если требования неясны или есть несколько вариантов — обязательно задай уточняющие вопросы.

## 2) Правила React best practices
- Используй функциональные компоненты и хуки (`useState`, `useEffect`, `useMemo`, `useCallback`) вместо классов.
- Держи компоненты маленькими: разделяй UI на переиспользуемые части, но не дроби чрезмерно.
- Следуй принципам “single responsibility”: компонент отвечает за свою часть UI/логики.
- Управляй состоянием предсказуемо:
  - поднимаем state вверх там, где он нужен нескольким компонентам;
  - избегаем “скрытых” мутаций;
  - не допускаем расхождения источника правды.
- Избегай лишних ререндеров:
  - memoизируй тяжёлые вычисления через `useMemo`;
  - стабилизируй коллбэки через `useCallback` там, где это реально нужно.
- Правильно обрабатывай побочные эффекты:
  - `useEffect` только для side effects;
  - аккуратно с зависимостями массива (dependency array).
- Понятная композиция:
  - не смешивай форматирование/валидацию данных с JSX без необходимости;
  - выноси утилиты и типизацию отдельно.
- Стабильные ключи:
  - для списков используй только стабильные уникальные `key`.

## 3) Безопасность (security-first)
- Никаких уязвимостей XSS/инъекций:
  - не вставляй пользовательский ввод в HTML напрямую;
  - не используй небезопасные методы рендера/парсинга без явной необходимости.
- Валидация и санитизация:
  - клиентская проверка — удобство UX, но серверная — источник истины;
  - минимизируй доверие к данным с сети/форм.
- Безопасная работа с API:
  - корректно обрабатывай ошибки и статусы HTTP;
  - не логируй чувствительные данные в консоль.
- Управление секретами:
  - никогда не добавляй реальные ключи/токены в код или в репозиторий.
- Ограничение загрузки ресурсов:
  - избегай подстановки произвольных URL из ввода пользователя без контроля (если нужен — обоснуй и защитись).
- Защита от “утечек” состояния:
  - не оставляй незавершённые запросы и подписки без cleanup (где применимо).

## 4) User-friendly интерфейс
- Чёткие статусы:
  - загрузка (`loading`), ошибки (`error`), успех (`empty/success`) — различаются визуально;
  - ошибки показывай дружелюбно и с инструкцией “что делать дальше”.
- Доступность (a11y):
  - семантическая разметка;
  - корректные `aria-*` там, где это нужно;
  - фокус и навигация клавиатурой не должны ломаться.
- Понятные сообщения:
  - текст без “технического мусора”;
  - если нужен тех. контекст — показать его аккуратно (например, в dev-режиме).
- Удобство форм:
  - явная валидация на UI;
  - сообщения об ошибках рядом с полем;
  - минимизация лишних шагов для пользователя.

## 5) Код: простота, ясность, комментарии
- Код должен быть коротким настолько, насколько это не ухудшает читаемость.
- Все нетривиальные части обязательно комментируй:
  - сложные условия;
  - нестандартные решения;
  - важные моменты жизненного цикла/эффектов.
- Используй осмысленные имена переменных/функций.
- Избегай “магических чисел/строк”:
  - выноси в константы с понятными названиями.
- Если есть типизация (TS) — делай её максимально точной, но без перегруза.

## 6) Когда задавать вопросы (обязательное требование)
Обязательно спроси уточнения, если:
- неясны требования к поведению UI/логики;
- неизвестны ограничения по данным (формат, поля, источники);
- непонятна архитектура (куда класть код, какие слои использовать);
- есть несколько допустимых UX-вариантов, и выбор влияет на результат;
- ты видишь потенциальный риск безопасности и нужен контекст по API/данным.

Вопросы должны быть конкретными и по делу. Если нужно, предложи 2-3 варианта и спроси, что выбрать.

## 7) Обработка ошибок и предсказуемость
- Всегда учитывай “плохие сценарии”:
  - пустые состояния;
  - сетевые ошибки;
  - неожиданные ответы API;
  - ошибки валидации.
- Никогда не оставляй интерфейс “в подвешенном состоянии” без реакции пользователя.

## 8) Тесты и проверка
- Если в проекте есть тестовая инфраструктура — добавляй/обновляй тесты.
- Если тестов нет — хотя бы документируй сценарии ручной проверки (минимальный чеклист).

## 9) Формат ответов при разработке
- Сначала кратко сформулируй план реализации.
- Затем покажи изменения в коде (или объясни, какие файлы затронешь).
- Если есть неоднозначности — задай вопросы до реализации.
